# Critical Multi-Agent Review — 2026-06-25

**Score: 68/100.** Method: 6 review lenses (literature, architecture, line-level detail, efficiency, weird-choices, tests), each followed by an adversarial verifier that re-checked every finding against the code and could reject it or add missed issues; then a synthesizer. 13 agents.

## Big-picture verdict
This is a genuinely above-average research repo: the numerically delicate core is correct and well-tested. RLS reproduces closed-form ridge to machine precision, the conceptor formula C=R(R+a^-2 I)^-1 and the leaky-ESN update match the literature, all five GEM/RWalk CL metrics (ACC/BWT/FWT/forgetting/intransigence) are independently re-derived as correct (verified at metrics.py:19-35), and the engine characterization goldens pin RNG order, W, spectral radius and full trajectories. METHODOLOGY.md is unusually self-aware and pre-empts most traps (symmetrization, upscaling, mock-data, leakage). The honest verdict, though, is that the package does NOT yet deliver the architecture its own design doc advertises, and that gap is the dominant theme across every lens. The single most consequential structural defect (confirmed independently by architecture, literature, weird, and detail lenses): the HEADLINE continual method — conceptors — is structurally excluded from the very benchmark/metrics harness that produces the paper numbers. ContinualBenchmark.run (benchmark.py:19-33) drives readouts via partial_fit/predict, but ConceptorClassifier exposes learn_class/predict and has no partial_fit, so BWT/forgetting are never computed for the method the package is built around. Compounding this, the conceptor classifier is positive-evidence-only with no normalization and no negative term, and omits the free-subspace incremental loading its own design doc (lines 81-90) specifies — leaving conceptor_and/conceptor_or as exported dead code (verified: zero references outside __init__/tests). So the conceptor 'continual mechanism' is really an independent per-class conceptor bank dressed up as Jaeger incremental loading. There is one real latent code bug (the Rayleigh-quotient spectral radius silently misreports rho for complex-dominant/non-normal W and ships a docstring claiming a 'norm check' that does not exist — connectome.py:457-491), but its blast radius is small: every realistic connectome path is symmetric (real spectrum) and never triggers it. The rest is a thick layer of doc-vs-code drift (Kalman/FORCE/SLDA/Replay/EWC advertised and cited but absent; sklearn TransformerMixin/BaseEstimator contract claimed but zero hits anywhere; ContinualBenchmark signature wrong; scenario param inert), plus genuine but minor robustness gaps (single/two-session split silently degenerate, several silent-empty/zero-result edge cases that should fail fast) and concentrated test blind spots on exactly the new product layer (CL algebra, LMS branch, validation guards, leakage-free split correctness). Net: the engine substrate is solid and trustworthy; the 'paper-grade continual-learning package' framing is currently aspirational — the scientific story is half-wired and the spec oversells the code. Eminently fixable, mostly via honest documentation downgrades plus a handful of small code changes, but as it stands a reader trusting the docs would believe in capabilities and a headline result that the code cannot produce.

## Confirmed issues (ranked)

### [major] Headline CL method (conceptors) cannot be driven by the benchmark/metrics harness — API split (partial_fit vs learn_class)
- **Where:** reservoirs/learning/benchmark.py:19-33 vs reservoirs/learning/continual.py:47,63
- **Fix:** Define one incremental-classifier protocol (e.g. partial_fit(X,y,classes)/predict) and make ConceptorClassifier conform — accumulating per-class state correlation across calls rather than recomputing per minibatch — so ContinualBenchmark can build an R matrix and report BWT/forgetting for conceptors. Verified: benchmark.run calls readout.partial_fit/predict; ConceptorClassifier has no partial_fit. This is the package's central structural defect.

### [major] Conceptor classifier omits free-subspace incremental loading AND negative evidence its own design doc specifies; AND/OR are dead code; positive-only evidence is scale-biased
- **Where:** reservoirs/learning/continual.py:26-32 (conceptor_and/or unused), 47-72 (learn_class/evidence/predict); docs/CONTINUOUS_LEARNING_DESIGN.md:81-90
- **Fix:** Verified: predict ranks by mean x^T C_k x only (no ||x||^2 normalization, no NOT(OR(others)) negative term), learn_class stores independent per-class conceptors (no free-subspace projection), and conceptor_and/conceptor_or have zero references outside __init__/tests. Either (a) implement combined evidence E_k = x^T C_k x + x^T NOT(OR_{m!=k} C_m) x and/or normalize by x^T x, and wire in free-subspace loading N=C AND NOT(OR previous); or (b) honestly redocument as an independent per-class conceptor bank, fix design-doc line 90 (drop '(neg)') and lines 81-83, and mark AND/OR experimental. Pick one; do not keep code and spec divergent. Scale-bias is demonstrable (a low-leak sample into a high-variance class misclassifies).

### [major-claimed, downgraded to minor by reachability (verifier consensus)] Power-iteration spectral radius silently wrong for complex-dominant/non-normal W; docstring claims a 'norm check' that does not exist
- **Where:** reservoirs/connectome.py:457-491 (_spectral_radius); duplicate reservoirs/_spectral.py:30-55
- **Fix:** Verified: convergence test is on the Rayleigh quotient lam=x^T A x; for a rotation/complex-dominant matrix it 'converges' to a wrong (often ~0) value so the eigvals fallback never fires, and W is then mis-scaled (ESP violation). The docstring's 'norm check below for complex-dominant cases' (line 463-464) is false — norm_y only normalizes x. BUT every realistic path is symmetric (graphmls undirected; nulls.py returns symmetric B; resize preserves symmetry) with a real spectrum, where the code is correct; no current path triggers it and no test exercises non-symmetric W. Fix: correct the false docstring, and use the iterate magnitude lam=||A x|| or fall back to eigvals on non-symmetric input. Do NOT switch the default symmetric path to exact eig blindly. Treat as latent robustness + docstring fix, not a live miscalibration.

### [minor] Single-session (and two-session) session-grouped split silently produces empty train set / ignores test_ratio
- **Where:** slither/data.py:159-166 (train_test_split_windows, group_by_session branch)
- **Fix:** Verified: with one unique session, n_test=max(1,int(1*0.25))=1 holds out the only session -> train_idx empty, no error (surfaces later as a zero-row ridge solve). With 2 sessions it always splits 50/50, silently ignoring test_ratio=0.25; 3->33%, 4->25%. Since session-grouped is now the leakage-free DEFAULT and the mock defaults to 3 sessions, this can quietly distort the headline metric. Raise a clear ValueError when len(uniq)<2 or the resulting train/test split is empty, and adopt a rounding policy (or warn) when the actual test fraction deviates materially from test_ratio.

### [minor (acknowledged limitation, understated)] Mock slither task is ~93% solvable from a single leaked input feature (prev-heading) with no reservoir

### [minor] RLS 'numerical stability' overstated: only re-symmetrization implemented, not the documented square-root/UD-factored P or state-noise injection

### [minor (latent — dormant utilities)] conceptor_and/or use raw double-pinv with no regularization in the singular case the design doc itself flagged; conceptor_from_states uses plain inv

### [minor (latent robustness)] rewire_degree_preserving zips edges with a separately-computed weight array of possibly-mismatched length, silently truncating with no guard
- **Where:** reservoirs/nulls.py:22-37
- **Fix:** Verified: G is built from (A!=0) including the diagonal then self-loops removed; weights come from strict upper-triangle triu_indices(N,1) (excludes diagonal). zip(G.edges(), weights) silently truncates to the shorter sequence — counts line up only for a symmetric zero-diagonal connectome (the stated invariant). On any input violating that (e.g. nonzero diagonal) it silently produces a wrong-density null. Add assert len(list(G.edges()))==len(weights) (or handle the diagonal explicitly) so it fails fast.

## Suggestions the adversarial verification REJECTED as overblown/wrong

- **EFF-1: eig->eigvals swap on random reservoirs is blocked by bit-for-bit goldens** — The verifier proved the premise BACKWARDS. The characterization goldens for dynamical snapshots (sr/W/states) use assert_allclose(rtol=1e-6, atol=1e-6) — verified at test_characterization.py:36-40 — NOT assert_array_equal. eig vs eigvals differ by ~1e-15, nine orders of magnitude inside tolerance, so the swap is safe and the 'must regenerate goldens' caveat is unnecessary. The 'bit-for-bit' language lives only in docstrings, not the test assertions. (Conversely, the float64->float32 EFF-6 change WOULD exceed tolerance — so the golden caveat is real there, for a different reason.) Either way both are premature at the package's actual scale (N in the hundreds).
- **BUG-1/F1 spectral radius bug is 'major' and corrupts current results** — Real code bug + genuinely false docstring, but severity is overstated for THIS repo. The verifier stress-tested 500 realistic asymmetric sparse connectome-like matrices (signed and unsigned) with 0/500 failures: any matrix with a dominant real (Perron) eigenvalue converges correctly. The failure needs a near-pure-rotation/normal-complex structure that connectome data never produces, and every shipped path symmetrizes first (real spectrum). Correctly downgraded from major to minor/latent. Fix the docstring and harden the fallback, but no current result is miscalibrated.
- **ARCH-02: nothing is a real sklearn estimator (won't pass check_estimator / Pipeline / clone) — major** — Factually true (zero sklearn hits) but downgraded: the design doc's section-3 header explicitly says 'API surface (sketch — final signatures TBD)', so the mixins are aspirational sketch, not a landed contract the code violates. For a research package not claiming Pipeline integration this is a doc-reconciliation issue, largely subsumed by ARCH-09, not a functional defect. Reduced to minor and folded into the doc-accuracy item.
- **ARCH-10 / EFF-3,4,5,6: missing OnlineReadout.fit means the 'joint upper-bound baseline cl_metrics requires' has no implementation; assorted O(N^3)/loop hot spots** — cl_metrics(R, joint=...) takes joint accuracies as a plain argument — verified at metrics.py:33-35 — it does NOT require any particular fit() method, so nothing is broken; a caller computes joint with connectome.fit or compute_wout. The efficiency items (conceptor pinv, benchmark T^2, slither preprocessing loops, dense float64) are one-time/construction costs or intrinsic to the metric definition (T^2 R-matrix is required by GEM), correctly self-rejected as premature. The only efficiency item with real teeth is EFF-2.
- **ARCH-05: 138-line ConnectomeReservoir constructor must be split into from_graphml/from_adjacency classmethods** — The description is accurate but the refactor is a matter of taste, not a defect. The adjacency= path is ALREADY first-class and clean in real usage (example 07 passes it as a simple kwarg), resize is already factored into _resize_adjacency, and classmethods would churn a goldens-pinned constructor for cosmetic gain. Low urgency.
- **L5: Maslov-Sneppen null destroys weight-degree coupling and is presented as 'the standard test'** — Technically accurate that uniform weight shuffling is a stronger randomization than strength-preserving nulls, but the disclosure gap is overstated: nulls.py:17 and METHODOLOGY.md:20 already state weights are reassigned (not strength-preserved), and Maslov-Sneppen itself is a binary-topology randomization so the citation is correct. A valid methodological nuance worth one doc sentence; a strength-preserving variant is optional future work, not a correctness issue.
- **TQ-9: conceptor eigenvalue '< 1.0' bound is structurally always-true** — Partially valid but the absolute framing was wrong: the verifier showed that at extreme apertures (1e9) floating-point r/(r+a^-2) can actually round to NOT < 1.0, so the bound is not literally always-true. Still a weak invariant worth strengthening to a quantitative pin, but a low-priority nit, and the 'cannot fail for any input' claim should be dropped.
